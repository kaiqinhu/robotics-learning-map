### 1.2 UMI 数据 + 真机数据融合

**问题：** UMI 数据（采集到的视频和接触信号，做过 SLAM 定位来标定位置）和真机带位置的数据怎么融合？

**背景理解：**
- UMI (Universal Manipulation Interface)：一种低成本数据采集范式——手持夹爪 + GoPro，用 SLAM 重建末端位姿
- UMI 数据量大、易采集，但精度低（SLAM 漂移）
- 真机数据精度高，但少、贵

**这是典型的：多源数据、不同质量的预训练问题**

**相关模块：**
- [M4-预训练范式](/M4-预训练范式) — 混合质量数据的预训练策略
- [M1-Riemannian-FM与SE3生成](/M1-Riemannian-FM与SE3生成) — UMI 的 SLAM 位姿在 SE(3) 上，真机位姿也在 SE(3) 上，但噪声性质不同

**可能的思路：**
```
方案 A — 数据统一：
  把 UMI 数据"升级"：用真机的运动学模型重新标注位姿
  但成本高，不实际

方案 B — 噪声建模：
  给模型输入加"数据来源tag"：[umi, low_quality] vs [real, high_quality]
  模型学会"对 UMI 数据中的位姿不太信"

方案 C — 两阶段训练：
  阶段 1：UMI 数据预训练（大量、噪声大）→ 学"大致会怎么动"
  阶段 2：真机数据微调（少量、精确）→ 学"精确怎么动"
```

**待研究：**
- [ ] 如何在同一个 FM 框架里处理不同质量的数据？
- [ ] 位姿标签的方差能否作为损失权重？高噪声 → 低权重

### 1.3 高频率小脑回路

**问题：** 高频率小脑回路怎么设计？触觉信号怎么输入？位姿和力控信号怎么输出？

**这是你整个设计里最工程化但也最关键的部分。**

**"小脑"的含义：**
```
大脑（~10 Hz）：  视觉理解、任务规划、全局策略
  ↓  目标位姿 + 力目标
小脑（~100-1000 Hz）：  实时伺服、力控、触觉反射
  ↓  力矩/关节命令
身体：  电机执行
```

**设计挑战：**

| 维度 | 挑战 |
|------|------|
| **频率** | 小脑必须 100-1000Hz，Transformer forward pass 做不到 |
| **触觉输入** | 触觉图像 30-60Hz，力信号 1000Hz，怎么对齐？ |
| **位姿+力联合输出** | SE(3) 和力目标耦合，不能分开算 |

**可能的架构：**

```
方案 A — 分层异步架构：

大脑（低频 10Hz，Transformer）:
  视觉(30Hz) + 任务 → Flow Matching → SE(3)轨迹 + 力目标

小脑（高频 1000Hz，轻量MLP/线性）:
  力信号(1000Hz) + 触觉(60Hz) + 大脑传来的目标
  → 阻抗控制器 → 关节力矩

  关键：小脑不做生成，只做"追踪"——把大脑的目标位姿+力，
       用经典阻抗控制去执行，同时用触觉做反射性修正


方案 B — 端到端但时间解耦：

FM 策略（中频 50Hz）:
  输入：视觉(30Hz) + 触觉(60Hz) + 力(降采样到50Hz)
  → 输出 SE(3) + 力目标序列

高频插值器（1000Hz）:
  → 把 FM 的 50Hz 输出插值到 1000Hz
  → 用 SE(3) 上的样条插值 (spline on SE(3))
```

**相关模块：**
- [C4-阻抗与导纳控制](/C4-阻抗与导纳控制) — 经典阻抗控制是小脑的理论基础
- [C2-SE3几何入门](/C2-SE3几何入门) — SE(3) 上的测地线插值
- → [I1-系统架构与部署](/I1-系统架构与部署) — 小脑回路设计

**待研究：**
- [ ] 小脑需要多快？（取决于任务：插销 1000Hz，抓取 100Hz 可能够）
- [ ] 触觉反射怎么融入？（检测到滑移 → 立即加力，不等大脑）
- [ ] 大脑和小脑之间怎么通信？(ROS2 topic? Shared memory?)

### 1.4 Flow Matching 加速

**问题：** Flow Matching 的加速怎么设计？

**背景：** FM 采样需要解 ODE，默认 Euler 法需要 N 步。加速方向：

```
加速方向 1 — 更少的 ODE 步数：
  Reflow（[C3-Flow-Matching基础](/C3-Flow-Matching基础)）
    → 多轮 reflow 后，路径变直，1-5 步就够
  
  蒸馏（Distillation）：
    用多步 FM teacher 生成 → 单步 student 直接输出 x₁

加速方向 2 — 更快的 ODE solver：
  Euler → RK4 → Dopri5（自适应步长）
  自适应步长：在路径弯曲处用小步，直的地方用大步

加速方向 3 — 模型架构加速：
  MoE 稀疏激活 → 减少每次 forward 的计算量
  量化（INT8）→ 加速矩阵乘法
  TensorRT / ONNX Runtime → 推理优化
```

**对你场景的特殊考虑：**

```
小脑回路场景（需要 100+ Hz）：
  FM 采样 N 步 → 每步一次网络 forward → 延迟 = N × forward_time
  
  如果 forward_time = 5ms, N = 10:
    延迟 = 50ms = 20Hz，不够！
  
  解决方案：
  → Reflow 把 N 压到 1-2
  → 或者：FM 只在大脑运行（10Hz）
  → 小脑用经典控制 + 触觉反射
```

**相关模块：**
- [C3-Flow-Matching基础](/C3-Flow-Matching基础#3-rectified-flow修正流) — Reflow 加速
- → [I1-系统架构与部署](/I1-系统架构与部署) — FM 推理加速

**待研究：**
- [ ] 你的场景需要多低的延迟？(夹爪力控 ~5ms？灵巧手 ~1ms？)
- [ ] Reflow 在 SE(3) + 力混合空间上还 work 吗？

## 二、千寻智能 — 后训练与泛化

### 2.1 门到门的泛化

**问题：** 如何从能打开一扇门，泛化到打开别的门？

**核心挑战：** 门的变化——尺寸、把手类型（旋转/按压/拉杆）、开门方向（推/拉）、铰链位置

**这本质上是：**
→ 视觉几何泛化（不同形状的门）
→ 力控泛化（不同阻力、不同开门力度）
→ 操作策略泛化（推 vs 拉 vs 旋转）

**相关模块：**
- [M4-预训练范式](/M4-预训练范式) — 预训练到新任务的泛化
- [M3-触觉表征与力预测](/M3-触觉表征与力预测) — 力控的泛化（不同门需要不同力曲线）

**可能的思路：**
```
方案 A — 任务分解 + 组合泛化：
  开门任务 → 子技能：接近把手 → 抓把手 → 施力 → 推/拉
  泛化 = 每个子技能独立泛化 + 重新组合

方案 B — Object-Centric 表征：
  不把门当成"像素"，而是"铰链 + 把手 + 面板"
  → 这三者的几何关系泛化

方案 C — 力控作为"硬知识"：
  视觉泛化难 → 但力控策略可能更容易泛化
  "推门"的力曲线（先大后小）在不同门上相似
```

**待研究：**
- [ ] 门泛化是否需要专门的测试基准？
- [ ] SE(3) 上的 FM 能否生成不同门对应的不同"开门弧线"？

## 三、核心洞察 — 位置 vs 力

> "位置轨迹，大家都还在做位置的预测，但其实核心在手上的精细力控呢？手眼力控、精细配合。"

### 为什么大家都在做位置预测？

1. **位置数据容易采集** — 遥操作、UMI、SLAM 都能给位置标签
2. **位置容易评估** — 位置误差好算，力用得"好不好"难定义
3. **位置是可见的** — 视觉上能看出位置对不对，力对不对看不出来

### 为什么力控才是核心？

```
实际操作中：
  位置对了 ≠ 操作成功
  
  插销：
    位置对着孔，但力用大了 → 卡住
    位置对着孔，但力太小了 → 没插进去
    
  抓取：
    位置在物体上，但夹持力不够 → 滑掉
    位置在物体上，但夹持力太大 → 捏碎
    
  → 位置是"必要条件"，力是"充分条件"
```

### 你的机会

```
现有工作：FM 生成位置轨迹 ✓，力控 ✗
你的方向：FM 联合生成 (SE(3) 位姿, 力轨迹)

这个联合生成的空间是：
  M = SE(3) × R^6

挑战：
  - 力数据难采集（需要 F/T 传感器，不是所有机器人都有）
  - 力的"好轨迹"难定义（什么力曲线是好的？）
  - 评估困难

机会：
  - 力控的生成模型是蓝海
  - 触觉+力觉结合 → 闭环力控
  - 灵巧手力控几乎没有好的生成方法
```
